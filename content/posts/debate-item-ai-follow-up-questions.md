## Why Debate Exists

Gather fails for a specific kind of question. If I need "email, phone, funding stage" — Gather is perfect. But if I need "understand how this founder thinks about their market" — Gather falls apart.

You can't regex-validate a market thesis. You can't pre-define the fields to extract because you don't know what the user will say. And asking one canned question ("What's your market thesis?") never gets you enough — you have to probe based on the answer, follow up on interesting threads, push on things that sound evasive.

That's the job of Debate. One starting question, then N follow-up questions generated by the LLM based on what the user actually said, then a summary that turns the whole exchange into structured outputs.

## The Debate Loop

![Debate item loop: starting question → response → follow-up → loop N times → summarize](/images/blog/debate-follow-up-loop.svg)

```python
class DebateItem(BaseAgendaItem):
    starting_question: str
    starting_question_is_dynamic: bool = False
    num_follow_ups: int = 3       # typically 3-5
    summary_schema: dict          # what to extract at the end

    follow_up_count: int = 0
    exchanges: list[dict] = []    # [{question, response}, ...]

    async def on_start(self):
        if self.starting_question_is_dynamic:
            question = await self.generate_starting_question()
        else:
            question = self.starting_question
        self.current_question = question
        self.emit(MessageEvent(content=question))

    async def on_user_response(self, message: str):
        self.exchanges.append({
            'question': self.current_question,
            'response': message,
        })
        self.follow_up_count += 1

        if self.follow_up_count >= self.num_follow_ups:
            return  # on_end will summarize

        # Generate the next follow-up based on what's been said
        next_q = await self.generate_follow_up()
        self.current_question = next_q
        self.emit(MessageEvent(content=next_q))

    @property
    def should_end(self) -> bool:
        return self.follow_up_count >= self.num_follow_ups

    async def on_end(self):
        outputs = await self.summarize_conversation()
        self.emit(EndAgendaItemEvent(
            item_id=self.id,
            outputs=outputs,
        ))
```

Three LLM calls happen inside one Debate item:
1. (Optional) Dynamic starting question generation
2. Follow-up generation after each user response
3. Final summarization that extracts structured outputs

Most Debate items run 3-5 follow-ups, so that's 4-6 LLM calls total per item. Not cheap. But Debate is used for high-value exchanges where the cost is justified — screening interviews, needs assessment, diagnostic conversations.

## Starting Question: Static vs Dynamic

The first question can be a literal string or an AI-generated opener. Most Debate items use static starting questions because the opening line sets the tone:

```yaml
type: debate
starting_question: "Tell me about the problem you're solving for customers."
num_follow_ups: 4
```

Dynamic starting questions are useful when the opening should acknowledge something the user just said in Q&A mode or in a prior agenda item:

```yaml
type: debate
starting_question_is_dynamic: true
starting_question_prompt: |
  Based on what the user said about their company ({{ company_name }})
  and their role, ask one open question about their go-to-market strategy.
  Keep it specific to their industry.
num_follow_ups: 4
```

The `{{ company_name }}` placeholder pulls from prior outputs (see [Day 5](/posts/convey-item-scripted-dynamic-messages/) for the placeholder mechanism). So if a prior Gather item collected the company name, the Debate opener uses it — the exchange feels personalized from the first line.

## Generating Follow-Ups

The interesting part of Debate is the follow-up generation. After every user response, the LLM sees:

1. The full exchange so far (Q1 + A1, Q2 + A2, ...)
2. A prompt describing what the Debate is trying to learn
3. Instructions for generating the next question

```python
async def generate_follow_up(self) -> str:
    system_prompt = f"""
    You are conducting an interview about: {self.topic}

    Goal of this conversation: {self.goal}

    Guidelines for your next question:
    - Probe deeper on the most interesting point in the user's last answer
    - Avoid yes/no questions
    - Don't repeat earlier questions
    - One question at a time
    - Keep it under 25 words

    Here is the exchange so far:
    {self.format_exchanges()}
    """

    response = await llm.chat(
        system_prompt=system_prompt,
        user_message="Generate the next follow-up question.",
    )
    return response.content.strip()
```

The `format_exchanges()` helper dumps prior Q/A pairs into a compact format:

```
Q1: Tell me about the problem you're solving.
A1: We help small businesses manage inventory across multiple sales channels.

Q2: How do you differentiate from existing players like Shopify or Square?
A2: We focus on the long tail — businesses too small for those platforms.
```

Each round, the prompt grows. By round 4, the LLM has seen 3 prior exchanges and can spot threads worth pulling on. "You mentioned the long tail — how do you acquire customers that small cheaply?" is exactly the kind of question a human screener would ask next.

## The Summary Step

At the end of the loop, Debate turns the whole exchange into structured outputs. This is where the item earns its keep — instead of dumping 5 paragraphs of conversation into a database field, it extracts named values.

```python
async def summarize_conversation(self) -> dict:
    schema = self.summary_schema  # e.g. {"problem_clarity": "int 1-5", ...}

    system_prompt = f"""
    Extract the following fields from this interview:

    {self.format_schema(schema)}

    If a field cannot be determined from the conversation, return null for it.
    Do not invent information not present in the transcript.

    Interview transcript:
    {self.format_exchanges()}
    """

    # Structured output — the LLM returns JSON matching the schema
    response = await llm.parse(
        system_prompt=system_prompt,
        user_message="Extract the fields as JSON.",
        response_model=schema,
    )
    return response.dict()
```

A summary schema for a founder screening Debate:

```yaml
summary_schema:
  problem_summary:
    type: string
    description: "One-paragraph summary of the problem the startup solves"
  market_size_estimate:
    type: string
    description: "Founder's estimate of the addressable market"
  differentiation:
    type: string
    description: "What makes this startup different from competitors"
  red_flags:
    type: array
    items: string
    description: "Specific concerns raised during the exchange"
  confidence_score:
    type: number
    description: "0-10 how confident the founder sounded about their market"
```

After summarization, those fields land in Redis's `:agenda:outputs` hash (from [Day 4](/posts/conversation-state-in-redis/)). The next agenda item can reference them. The completion webhook fires with them. A CRM entry gets populated with clean structured values — not a 2000-word transcript.

This is the "turn mess into data" step that makes Debate usable for real business workflows.

## What Makes a Good Debate Prompt

The prompts that drive Debate follow-ups matter more than any other part. I'll share the patterns that worked:

**Give the LLM a clear goal.** "You're screening founders for a venture fund" or "You're helping a user describe their technical requirements" — not "ask some questions." The goal shapes what counts as interesting enough to follow up on.

**Tell it what to avoid.** "No yes/no questions. Don't repeat earlier questions. One question at a time." These sound obvious but the LLM will violate all of them unless told not to.

**Include the "why" of each question.** When the summary prompt says "extract confidence_score — how confident the founder sounded about their market" — it nudges the follow-ups to probe confidence during the exchange. The summary schema becomes an implicit guide for the follow-ups.

**Limit the round count.** 3-5 is the sweet spot. Under 3 is too shallow — the LLM never gets to the interesting stuff. Over 5 starts to feel interrogative and users bail. 4 is my default.

## When Gather and Debate Appear Together

Most interesting agendas use both. A typical founder screening looks like:

```
Convey    → "Thanks for applying. I'll ask a few questions."
Gather    → name, email, startup_name, funding_stage
Debate    → "Tell me about the problem you're solving" (4 rounds)
Debate    → "Walk me through your go-to-market strategy" (4 rounds)
Q&A       → "Any questions about our program?"
Convey    → personalized closing (dynamic)
```

Gather handles the structured fields. Debate handles the qualitative assessment. Convey frames the experience. The state machine from Day 1 ties them together.

## Cost Considerations

Debate is the most expensive item per turn:
- 1 LLM call for starting question (if dynamic)
- 1 LLM call per follow-up × N rounds
- 1 LLM call for summary

At 5 rounds with dynamic starting, that's 7 LLM calls in one item. For a 6-item agenda with 2 Debate items (10 rounds each), I'm looking at 15-20 LLM calls per conversation.

This is why I don't over-use Debate. Gather and Convey handle 90% of the conversation flow cheaply; Debate is reserved for the moments where open-ended exploration actually pays off.

## Closing

Debate is the item that makes the platform replace dedicated screening tools. A founder screening interview used to cost 30 minutes of a partner's time. The Debate item doesn't replace the partner, but it collects enough structured signal that the partner can review a one-page summary in 60 seconds and decide whether to spend the 30 minutes.
